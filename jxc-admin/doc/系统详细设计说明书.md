## 1 引言

### 1.1 编写目的
本文档对「进销存管理系统」重构版进行详细设计说明，覆盖系统总体架构、技术选型、数据库设计、后端接口设计、前端模块设计、核心业务流程与部署方案，作为开发、测试、部署与后续维护的依据。

### 1.2 项目背景
原系统（jxc-parent）基于 Spring Boot 2.2 + MyBatis-Plus + Shiro + jQuery/EasyUI 多页 HTML 实现，但后端 Controller 仅为代码生成器产出的空壳（无业务实现），前端为传统 jQuery + EasyUI 页面，前后端耦合于同一工程。本次重构在**保持数据库结构不变**的前提下，采用「前后端分离」架构全面重写，后端使用 Spring Boot 2.7 提供 RESTful API，前端使用 Vue 3 + Element Plus 构建 SPA 单页应用。

### 1.3 系统目标
- 实现进货管理、销售管理、库存管理、统计报表、基础资料、系统管理六大核心业务模块。
- 自动维护商品库存（入库/出库/报损/报溢/退货联动库存）。
- 基于角色的菜单权限控制（RBAC）。
- 提供商品采购/销售/供应商/客户等多维度统计分析与可视化图表。

## 3 数据库设计

数据库名 `jxc`，字符集 utf8mb4。**本次重构完全沿用原数据库结构，未做任何表结构调整**，共 23 张表，分四类：

### 3.1 系统管理表（5 张）

| 表名 | 说明 | 主要字段 |
|------|------|----------|
| `t_user` | 用户 | user_id(PK)、user_name、password、true_name、remarks |
| `t_role` | 角色 | role_id(PK)、role_name、remarks |
| `t_user_role` | 用户-角色 | user_role_id(PK)、role_id(FK)、user_id(FK) |
| `t_menu` | 菜单 | menu_id(PK)、menu_name、p_id(父菜单)、menu_state、menu_url |
| `t_role_menu` | 角色-菜单 | role_menu_id(PK)、role_id(FK)、menu_id(FK) |

权限模型为标准 RBAC：用户 → 角色 → 菜单（多对多）。菜单为树形结构（p_id，根节点 p_id=-1）。

### 3.2 基础资料表（4 张）

| 表名 | 说明 | 主要字段 |
|------|------|----------|
| `t_supplier` | 供应商 | supplier_id(PK)、supplier_name、contacts、phone_number、address |
| `t_customer` | 客户 | customer_id(PK)、customer_name、contacts、phone_number、address |
| `t_goods_type` | 商品类别（树形） | goods_type_id(PK)、goods_type_name、p_id、goods_type_state |
| `t_goods` | 商品 | goods_id(PK)、goods_code、goods_name、inventory_quantity、min_num、purchasing_price、selling_price、goods_unit、goods_type_id(FK)、state |
| `t_unit` | 商品单位 | unit_id(PK)、unit_name |

### 3.3 业务单据表（12 张，6 对「主表 + 明细表」）

每类单据由一张**主表**（单头：单号/金额/日期/操作员）和一张**商品明细表**（商品快照：编码/名称/数量/单价/金额）组成。明细表冗余存储商品信息，保证历史单据可追溯。

| 业务 | 主表 | 明细表 | 库存影响 |
|------|------|--------|----------|
| 进货入库 | `t_purchase_list` | `t_purchase_list_goods` | **库存 +** |
| 采购退货 | `t_return_list` | `t_return_list_goods` | **库存 −** |
| 销售出库 | `t_sale_list` | `t_sale_list_goods` | **库存 −** |
| 客户退货 | `t_customer_return_list` | `t_customer_return_list_goods` | **库存 +** |
| 商品报损 | `t_damage_list` | `t_damage_list_goods` | **库存 −** |
| 商品报溢 | `t_overflow_list` | `t_overflow_list_goods` | **库存 +** |

### 3.4 日志表（1 张）

| 表名 | 说明 | 主要字段 |
|------|------|----------|
| `t_log` | 系统操作日志 | log_id(PK)、log_type、content、log_date、user_id(FK) |

### 3.5 关键设计要点
- 所有单据明细采用**商品快照冗余**（goods_code/goods_name 等存入明细），避免商品改名/删除影响历史单据。
- `t_goods.state`：0 初始、1 已入库、2 已有进销单据（不可删除）。
- 金额字段使用 `float`（沿用原设计，生产环境建议升级为 `decimal`）。
- 单据与供应商/客户/用户通过外键关联，保证数据完整性。

## 4 后端详细设计

### 4.1 分层架构
严格遵循 **Controller → Service → Mapper** 三层架构：
- **Controller**：仅负责接收请求、参数校验、调用 Service、返回统一 `Result` 包装。
- **Service**：承载全部业务逻辑（事务控制、库存联动、日志记录）。
- **Mapper**：继承 MyBatis-Plus `BaseMapper`，复杂查询使用 XML。

### 4.2 统一响应格式
所有接口返回统一 JSON 结构：
```json
{
  "code": 200,        // 200 成功，500 失败
  "message": "success",
  "data": {}          // 业务数据
}
```
- 成功：`Result.success(data)`
- 失败：`Result.error(message)`，由 `GlobalExceptionHandler` 统一捕获 `BusinessException` 及未知异常。

### 4.3 认证与权限

**登录流程**：
1. 前端 POST `/api/auth/login`（userName、password）。
2. 后端校验用户名密码，生成 JWT（载荷含 userId、userName，有效期 24h），返回 token + 用户菜单树。
3. 前端将 token 存 localStorage，后续请求头携带 `Authorization: Bearer <token>`。
4. `JwtInterceptor` 拦截除 `/auth/**` 外所有请求，解析 token 并将当前用户存入 `CurrentUser`（ThreadLocal）。
5. 菜单权限：admin 用户拥有全部菜单；普通用户通过 `t_user_role → t_role_menu → t_menu` 联查得到其授权菜单树。

**权限粒度**：菜单级（前端动态渲染侧边栏菜单 + 路由），接口级（JWT 拦截器统一鉴权）。

### 4.4 核心 REST API 清单

> 所有接口前缀 `/api`，下表省略前缀。

#### 认证
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/auth/login` | 登录，返回 token + 菜单 |

#### 基础资料
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/supplier/list` | 供应商列表 |
| POST | `/supplier` | 新增/修改供应商 |
| DELETE | `/supplier/{id}` | 删除供应商 |
| GET/POST/DELETE | `/customer/*` | 客户管理（同上） |
| GET | `/goods/list?typeId=&codeOrName=` | 商品列表（带类别名） |
| GET | `/goods/alarm` | 库存报警商品 |
| POST | `/goods` | 新增/修改商品 |
| DELETE | `/goods/{id}` | 删除商品（有单据的禁止删） |
| GET | `/goodsType/tree` | 商品类别树 |
| GET/POST/DELETE | `/unit/*` | 商品单位 |

#### 业务单据（以进货为例，其余结构一致）
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/purchase/list?supplierId=&state=&startDate=&endDate=` | 进货单查询 |
| GET | `/purchase/{id}` | 进货单明细（含商品列表） |
| POST | `/purchase` | 新增进货单（事务：写主表+明细+加库存+写日志） |
| PUT | `/purchase/settle/{id}` | 结算 |

> 销售类（`/sale`）、客户退货（`/customerReturn`）、采购退货（`/return`）、报损（`/damage`）、报溢（`/overflow`）接口结构一致。

#### 系统管理
| 方法 | 路径 | 说明 |
|------|------|------|
| GET/POST/DELETE/PUT | `/user/*` | 用户管理（含角色分配、改密） |
| GET/POST/DELETE | `/role/*` | 角色管理（含菜单分配） |
| GET | `/menu/list` | 菜单列表 |
| GET | `/log/list` | 系统日志查询 |

#### 统计报表
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/count/purchase` | 商品采购统计 |
| GET | `/count/sale` | 商品销售统计 |
| GET | `/count/supplier` | 供应商采购统计 |
| GET | `/count/customer` | 客户销售统计 |
| GET | `/count/saleDay` | 按日销售统计 |
| GET | `/count/saleMonth` | 按月销售统计 |

### 4.5 关键业务逻辑

**进货入库事务（PurchaseListService.savePurchase）**：
1. 生成单号（`JH` + 时间戳），写入主表（含操作员=当前登录用户、日期）。
2. 遍历明细：计算金额，写入明细表；**更新商品库存（+=数量）、更新采购价、标记 state=2**。
3. 累加应付金额，更新主表。
4. 写入系统日志。

所有单据保存均使用 `@Transactional` 保证主表、明细、库存、日志的原子性。退货/销售类逻辑对称（减库存）；报损减库存、报溢加库存。

## 5 前端详细设计

### 5.1 整体设计
- 单页应用（SPA），Vue Router Hash 模式。
- `vite.config.js` 配置开发代理：`/api` → `http://localhost:18080`，解决跨域。
- Axios 封装统一注入 token、统一错误提示、登录失效自动跳转。
- 后端返回 `{code, message, data}`，响应拦截器统一解包。

### 5.2 路由与权限
- 全局前置守卫：无 token 跳转 `/login`。
- 登录后根据返回的菜单树动态渲染侧边栏（`layout/index.vue` 递归 `el-menu`），路由按 `menuUrl` 映射。
- 菜单未授权的项不渲染，实现菜单级权限控制。

### 5.3 页面清单（29 个）

| 模块 | 页面 | 功能 |
|------|------|------|
| 首页 | Home | 数据看板（报警数/商品数） |
| 基础资料 | Supplier / Customer | 供应商/客户 CRUD |
|  | Goods | 商品管理（类别树+商品列表+弹窗编辑） |
|  | Stock | 报损/报溢记录查询（Tab） |
| 进货管理 | Purchase / Return | 进货入库/采购退货录入 |

|  | SaleSearch / ReturnSearch | 单据查询+明细弹窗 |
| 库存管理 | Damage / Overflow | 报损/报溢录入 |
|  | Alarm | 库存报警列表 |
|  | Search | 当前库存查询 |
| 统计报表 | Supplier / Customer | 供应商/客户统计表格 |
|  | Purchase / Sale | 商品采购/销售统计（表格+柱状图） |

### 5.4 通用交互模式
- **列表页**：搜索栏 + 表格 + 分页（数据量小用前端全量）+ 操作列（编辑/删除）。
- **录入页**：表头选择（供应商/客户）+ 商品多选弹窗 + 可编辑明细表格（数量/单价实时算金额）+ 合计行。
- **统计页**：日期范围 + 表格 + ECharts 图表（柱状/折线）。

## 6 系统部署与运行

### 6.1 环境要求
- JDK 1.8+（已在 JDK 21 验证编译）
- Maven 3.6+
- Node.js 16+（已用 Node 22 验证构建）
- MySQL 5.7+/8.0

### 6.2 数据库初始化
```bash
mysql -uroot -p < jxc-admin/src/main/resources/db/jxc.sql
```
默认账号：`admin / admin123`。

### 6.3 后端启动
```bash
cd erp-jxc/jxc-backend
# 修改 src/main/resources/application.yml 中数据库账号密码
mvn spring-boot:run
# 服务启动于 http://localhost:18080/api
```

### 6.4 前端启动
```bash
cd erp-jxc/jxc-frontend
npm install
npm run dev      # 开发模式 http://localhost:5173
npm run build    # 生产构建，产物在 dist/
```

### 6.5 访问
浏览器打开 `http://localhost:5173`，使用 `admin/admin123` 登录。

## 7 重构前后对比

| 维度 | 原系统（jxc-parent） | 重构版（erp-jxc） |
|------|---------------------|---------------------|
| 架构 | 前后端耦合（JSP/webapp） | 前后端分离（REST + SPA） |
| 后端业务 | Controller 空壳，无业务实现 | 完整三层架构，全部业务落地 |
| 安全 | Shiro + 图形验证码 | JWT 无状态鉴权 |
| 前端 | jQuery + EasyUI 多页 | Vue 3 + Element Plus 单页 |
| 体验 | 跳转刷新、交互老旧 | SPA 无刷新、组件化、响应式 |
| 图表 | ECharts 嵌入 | ECharts 组件化集成 |
| 代码质量 | 代码生成骨架 | 手写分层、统一异常/响应 |
| 可维护性 | 低 | 高（模块清晰、职责单一） |

### 9.3 已知限制与后续优化建议
- 密码明文存储，建议引入 BCrypt 加密。

- 无分页（数据量小），数据增长后建议补充分页查询。
- 统计接口可增加缓存（如 Redis）提升大数量下的响应速度。
- 前端大 chunk 可做 manualChunks 拆分优化首屏加载。
